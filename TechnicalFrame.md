# 《行旅》技术框架建议

## 1. 项目技术特点（来自 GDD）

- **纯文字 UI**：叙事框、状态框、物证/线索/卷轴/感应/交互等框，无 2D/3D 渲染。
- **骨架 + 血肉**：策划数据（核心剧情导向、禁忌、状态公式，均可选）驱动流程；AI 负责描述、选项文案、异史凝练。
- **重状态与逻辑**：命烛/根脚/鉴照、多种「害」、物证/线索、结局矩阵，需要清晰的状态机与数据表。
- **平台**：Windows 优先（可后续扩展其他平台）。

因此技术选型应**优先迭代速度与叙事/数据可维护性**，不必引入重型游戏引擎。

---

## 2. 推荐技术栈

### 2.1 方案 A：Web 技术栈（推荐）


| 层级        | 选型说明                                                                       |
| --------- | -------------------------------------------------------------------------- |
| **运行壳**   | **Electron** 或 **Tauri**：桌面 exe，内嵌网页。Tauri 更轻、体积小；Electron 生态更熟、调试方便。      |
| **前端**    | **React** 或 **Vue 3**：组件化实现「叙事框 / 状态框 / 感应框」等，状态用 React Query / Pinia 等管理。 |
| **样式**    | **CSS/Tailwind**：文字排版、分栏、高亮（鉴照关键词）、简单动效（如选项出现）。                            |
| **逻辑与数据** | **TypeScript**：类型安全，便于维护节点 ID、状态结构、指令包。                                    |
| **AI 调用** | **Node 端或前端**：通过 HTTPS 调用 OpenAI/Claude/本地 LLM API；敏感 API Key 放环境变量或本地配置。  |


**优点**：UI 迭代快、热重载、易做存读档（序列化 state + 历史）、跨平台简单。  
**缺点**：需打包成桌面应用；若未来要做移动端，可再考虑同一套逻辑 + 不同壳。

## 4. 核心模块职责

### 4.1 Skeleton Loader（骨架加载）

- **输入**：JSON/YAML 或数据库（界、事件、剧情点、节点树、Plot_Guide/核心剧情导向、禁忌、状态公式、门禁条件）。
- **输出**：内存中的「当前节点上下文」，供 Narrative Engine 与 AI Bridge 使用。
- **建议**：按「界」或「事件」分文件，用 `Node_ID` 等稳定 ID 引用，便于策划与版本管理。

### 4.2 Narrative Engine（叙事引擎）

- **职责**：根据玩家选择推进节点；执行门禁判定（物证/线索/状态）；计算命烛/根脚/鉴照与害的增减；在适当时机触发「结案」「归档异史」；检查结局条件并跳转结局。
- **与 GDD 对应**：入界、感应、抉择、结案与归档、状态演算、剧情门禁、因果收束。

### 4.3 AI 叙事引擎（`aiEngine/` + `aiBridge`）

> 完整设计见 **GDD 5.5**。当前仓库已落地分层实现（非雏形）：`game/src/game/aiEngine/`（`chat`、`dataAcquisition`、各场景 `prompts`），`aiBridge.ts` 为对外兼容入口；细则与完成度见根目录 **TODO.md**。

- **数据获取**：从骨架、GameState（三相+害）、物证、线索、卷轴、抉择历史、`NarrativeContextManager` 摘要聚合上下文。
- **数据修改**：数值与门禁仍以骨架 + `GameState.applyChoice` 为准；AI 主要改**展示层**叙事与补念文案；异史经凝练后写入卷轴条目。
- **叙事上下文管理**：事实条数/字数上限；灵损只劣化注入模型的摘要；鉴照混浊等进 prompt。
- **场景化提示词**：主叙事、异史、补念等模板在 `prompts/`；叙事含 plot_guide 校验重试与禁忌重试；补念在叙事成稿后请求并带入【当前境遇正文】。
- **与策划内容的交互**：门禁、`violatesTaboo`、选项后果（state、next、conclusion）由骨架与状态机控制。
- **Engine v2（即时生成 + 降级）**：`public/data/design-seed.json` 供 `runPlanner` 生成 `StoryOutline`；每拍 `runDirector` 产出 `NodeDirective`；`contextAssembly` 分层语境（L0–L5）；`generateDynamicBeatNarrative` / `generateDynamicBeatChoices` 驱动 `dyn:{realm}:{beat}` 节点；`WorldStateGraph` 写入 `GameState.worldGraph` 并随存档 **SaveData v3** 持久化。Planner/API 失败时保持 **骨架遍历**（`engineMode: skeleton`）。

**境遇正文（玩家读到的具体句子）从哪里来？**

| 节点类型 | 正文来源 | 是否会在 `prologue.json` 里出现那句原文？ |
|----------|----------|-------------------------------------------|
| **骨架节点**（非 `dyn:`，有 `plot_guide` 等） | 运行时调用 `generateNodeNarrative`，模型在 **骨架 `description` / `story_beat` / plot_guide** 约束下重写；无 Key 时用 JSON 里的 `description`。 | **可能**：流水线生成的 `description` 会写进 JSON；AI 当场写的成稿**不一定**与文件里旧稿一致。 |
| **动态拍**（`dyn:界id:拍号`，Engine v2） | **完全由当时那次 API 调用生成**：输入为 `design-seed`、`StoryOutline` 当前拍摘要、锚点 `must_include`、`runDirector` 的 directive、世界图摘要、玩家状态行等（见 `contextAssembly` + `buildDynamicBeatNarrative`）。 | **一般不会**：像「旁边躺着一只干涸的墨水瓶…」这类细描写是模型即时发挥，**本地文件只约束导向词/禁忌/拍摘要，不写这句全文**。成稿会写回运行时节点并可能进存档，但**不等于**仓库里预先存在该句。 |

若要「每一句都在自己掌控的文档里」，需要：**关掉 v2 只走骨架**，或把该界做成纯骨架节点并主要依赖 `prologue.json` / `skeleton.json` 的 `description`，而不是 `dyn:` 动态拍。

### 4.4 State Manager（状态管理）

- **三相**：命烛、根脚、鉴照。**UI 仅显示档位（旺盛/摇曳/熄灭 等），不显式展现百分比**。
- **害**：列表或集合，记录当前生效的害类型与强度（内部用）。**UI 仅显示害名称，不展现强度**。
- **物证与线索**：列表结构；支持「凭物感应」「语义解锁」等查询。
- **卷轴**：已生成异史条目列表，带标签（真史/疑伪/秽等）供结局与后续节点使用。

### 4.5 UI 与存读档

- **各框**：对应 GDD 第 6 节，用组件拆分；通过全局 state（如 Zustand/Pinia）与 Game Loop 同步。
- **存读档**：将 State Manager 的完整状态 + 当前节点 + 历史选择序列化，写入**数据库**（见 GDD 5.6）或 IndexedDB；读档时反序列化并恢复 Game Loop。

## 5. 数据格式建议（与 GDD 5.3 对齐）

**节点指令包（给 AI）示例结构：**

```json
{
  "Node_ID": "ZheJiYuan_02",
  "Plot_Guide": ["焚烧坑", "草灰涂面", "禁忌：不可回头"],
  "State_Filter": { "ling_sun": 40, "shou_chao": 15 },
  "Taboo": ["不可回头"],
  "Objective": "结案：获得异史线索【焚尸者言】"
}
```

**骨架数据（策划维护）建议包含：**

- 节点元数据：`Node_ID`、所属事件/剧情点、后继节点或分支条件。
- 每个节点的 `Plot_Guide`（核心剧情导向，可选）、`Taboo`、`Objective`、门禁（物证/线索/状态条件）。
- 状态公式表：动作或事件对命烛/根脚/鉴照及害的增减规则。
- 结局表：结局 ID、触发条件（卷轴比例、状态阈值、线索集合等）。

## 6. 技术风险与注意点

- **AI 延迟与错误**：前端需 loading 与超时处理；对关键节点可做「重试」或「降级到策划兜底文案」。
- **API Key**：不要写死在前端代码中；用环境变量或本地配置文件，并加入 .gitignore。
- **内容量**：先用一个界、少量害与结局跑通全流程，再扩展数据与提示词，避免一次性铺太大导致难以调试。

---

## 7. 小结

- **推荐以 Web 技术栈（Electron/Tauri + React/Vue + TypeScript）** 为主：开发效率高、易维护叙事与状态逻辑、便于与 AI API 集成。
- **架构上**：Skeleton Loader、Narrative Engine、AI Bridge、State Manager 分离，使「策划数据」「玩法逻辑」「AI 调用」边界清晰，便于协作与测试。
- **数据与 GDD 一致**：节点指令包、骨架中的 Plot_Guide/禁忌/目标与 GDD 5.3 保持一致；数据存储见 GDD 5.6，需搭建数据库。
