# 体验优化指南

> 硬性开发主干已齐，接下来以「玩 → 记问题 → 定点改 → 再玩」为主。本指南帮你**知道该打开哪个文件、改什么、怎么验证**。

---

## 一、推荐工作方式（别一次改太多）

1. **先完整跑一条路径**（见下方「首跑检查清单」），用手机或备忘录记下：**哪一步卡、哪句文案怪、哪块 UI 别扭**。
2. **每次只动一类事**：例如「只调主叙事 prompt」或「只改加载提示文案」，避免一次改五个文件，出问题不好归因。
3. **改完立刻 `npm run dev` 复现同一条路径**，确认变好再提交。
4. **可选**：`npm run test:unit` + `npm run test:ai` 确保没碰坏规则类逻辑（禁忌、plot_guide、结局判定等）。

---

## 二、首跑检查清单（建议按顺序勾）

在浏览器打开 `http://localhost:5173`（或 `npm run electron:dev`），准备一把可用的 API Key（或先「跳过骨架游玩」只看骨架）。

- [ ] **首屏**：无 Key 门闸文案是否好懂？保存 / 跳过是否顺手？
- [ ] **加载**：首屏「加载中…」、Planner「正在构思故事大纲…」是否过长或过短、是否需要补充说明？
- [ ] **序章**：境遇正文是否 1–2 句点题？`*` / `[]` 高亮在鉴照各档位下是否可读？
- [ ] **感应**：正文未出时「待正文落定…」是否清晰？选项是否有一条明显贴骨架？
- [ ] **物证 / 线索**：获得条带 `[书箱一沉…]` / `[心头一亮…]` 是否显眼、时长是否合适？
- [ ] **卷轴 / 异史**：凝练文风、标签（真史/疑伪/秽）是否符合预期？
- [ ] **结案 / 结局**：系列标题与描述是否读得通？多界时「前往下一界」是否出现且可点？
- [ ] **Planner 失败**：是否出现顶部 Toast？文案是否够指引（网络 / Key / 额度）？
- [ ] **交互菜单**：换界、存档槽、小屏「卷轴」入口是否顺手？

把**具体界面 + 原话/截图描述**记下来，再对照第三节找文件。

## 四、你「不知道如何下手」时的最小三步

1. **只回答一个问题**：当前最刺眼的，是 **叙事**、**选项**、**UI 字**、**节奏数据** 里的哪一种？
2. **打开上表对应的一个文件**，用编辑器搜索关键词（如「禁止」「鉴照」「感应」）。
3. **改一小段**，保存 → 刷新游戏 → 同一路径再走一遍。

若刺眼的是「某一句具体 AI 输出」，先判断：是**每次都这样**（改 prompt / 模型）还是**偶尔**（可加重试条件，见 `generateNodeNarrative` 内 plot / taboo 重试）。

## 五、与策划文档的关系

- **文风、节点逻辑、流水线约定**：[`design/AI功能设定/`](../design/AI功能设定/) 下 `ai1_outline.md`、`ai2_nodes.md`、`ai3_texts.md` —— 改大方向时可先改 md，再决定是否重跑 `npm run generate:prologue`（耗 API）。
- **只改已有节点成稿、不重跑 AI**：改 `generated/chapters/prologue/texts.json` 后 `node scripts/merge-only.mjs prologue`（见 [`README.md`](../README.md)）。

---

## 六、建议优先级（空闲有限时）

1. **主叙事 + 感应**（玩家停留最久）→ `narrative.ts`、`choices.ts`  
2. **加载与失败提示**（减少「是不是坏了」焦虑）→ `App.tsx`、`plannerUi.ts`  
3. **异史卷轴**（收尾体验）→ `yishi.ts`  
4. **骨架关键节点**（序章/折戟原转折处）→ JSON 手改
