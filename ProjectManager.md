# 《行旅》项目管理建议

## 1. 项目特点简要归纳

- **类型**：纯文字叙事 + AI 生成，无图形渲染，无复杂物理。
- **核心风险**：叙事质量依赖「骨架 × AI」的协同；内容量大（多界、多结局、大量害/状态）。
- **团队假设**：小团队或独立开发，策划与程序紧密协作。

因此更适合**轻量、内容可追踪、AI 迭代可控**的管理方式，而不是重型瀑布或大型 Scrum。

---

## 2. 推荐方法论：轻量 Agile + 内容管线

### 2.1 总体思路

- **功能开发**：按「可玩里程碑」拆成小迭代（约 2–4 周一轮），每轮交付可运行、可测试的版本。
- **内容生产**：与程序解耦，用「骨架数据 + 配置」驱动，版本与 Git 同步，便于回溯和协作。
- **AI 调优**：单独成「AI 迭代」周期（提示词、指令包格式、约束规则），与玩法迭代交错进行。

不追求严格 Scrum（不强制固定 sprint 长度、角色），以**可演示、可试玩**为主要验收标准。

### 2.2 迭代节奏建议

| 阶段       | 目标                         | 周期参考   |
|------------|------------------------------|------------|
| 原型验证   | 单界、单剧情线、基础感应 + AI 回包 | 2–3 周     |
| 核心循环   | 命烛/根脚/鉴照 + 3–5 种害生效 | 3–4 周     |
| 内容扩展   | 多界、多事件、物证/线索系统   | 持续 4–8 周 |
| 结局与收束 | 结局矩阵实现与条件校验       | 3–4 周     |
| 打磨与发布 | 存读档、性能、叙事一致性     | 2–4 周     |

## 3. 工具与协作

### 3.1 任务与进度

- **任务看板**：GitHub Projects / Trello / Notion 任选其一，列出「功能 / 内容 / AI / Bug」。
- **优先级**：每轮迭代前定 3–5 个「本轮必完成」项，其余放 backlog。
- **验收**：功能以「可运行 + 符合 GDD 某条描述」为准；内容以「骨架能正确驱动 AI 且可复现」为准。

### 3.2 内容与数据

- **骨架与配置**：全部用**文本格式**（如 JSON/YAML）存放，纳入 **Git**。
  - 便于 diff、merge、回滚，也便于策划直接改节点、锚点、禁忌。
- **命名规范**：界/事件/剧情点/节点 ID 统一（如 `ZheJiYuan_02`），便于代码和 AI 指令包引用。
- **大文档**：GDD、设定表、结局条件表建议用 Markdown 或结构化文档，与仓库同源，避免散落多处。

### 3.3 AI 相关

- **指令包与提示词**：单独目录版本管理（如 `prompts/`、`instruction_templates/`），每次改 prompt 或 JSON 结构都提交，便于复现和 A/B 测试。
- **测试用例**：对关键节点保存「输入状态 + 期望约束」（例如必须出现某真理关键词），用于回归测试 AI 行为是否越界。

### 3.4 Prompt 分析体系：分阶段，核心循环后再上量

**结论：先不建完整的「Prompt 好坏量化体系」，等核心循环（入界 → 感应 → 抉择 → 结案）跑通、能稳定玩通一条线后再迭代。**

- **现在（M0–M1）**：只做「版本化 + 人工感受」。每次改 prompt 打 tag 或记在 commit 里，玩几把凭体感判断是否更好即可。避免为指标建一堆脚本、反而拖慢验证节奏。
- **核心循环跑通后（M1 尾 / M2）**：加**最小可量化**层。
  - **日志**：每次请求记录「节点 ID、指令包版本、是否含真理关键词、是否触犯禁忌、响应长度」等少量字段，落 JSON/CSV。
  - **通过率**：对关键节点用既有「测试用例」跑 N 次，算「满足约束次数 / N」作为简单通过率，用于回归（改完 prompt 别把原来能过的搞崩）。
- **再往后（M3+，可选）**：若发现叙事质量成瓶颈、需要系统对比不同 prompt 版本时，再上**结构化评估 + A/B**。
  - **数据**：同一批测试用例/固定种子，跑 A 版与 B 版 prompt，输出结构化结果（如是否含锚点、禁忌违反次数、异史长度等）存成表格。
  - **工具**：用 **Python + Pandas** 读 CSV/JSON，做描述统计、通过率对比、简单显著性（如比例差的置信区间或卡方）；需要时可做 A/B 两组的分布对比。此时再考虑「如何量化 Prompt 好坏」（例如：锚点命中率 + 禁忌不触犯率 + 主观打分取平均）。
  - 不必一开始就上复杂统计或自动化流水线，等真有多次 prompt 版本需要对比时再加。

这样既保留「日后用数据 + Pandas 做 A/B、量化好坏」的扩展点，又避免在还没跑通核心循环时把体系做复杂。

## 4. 里程碑与交付物（建议）

| 里程碑           | 交付物 |
|------------------|--------|
| M0 原型          | 单界进入 → 若干节点 → 感应选项 → AI 回包 → 结案生成一条异史；可本地运行。 |
| M1 状态与害      | 命烛/根脚/鉴照增减与表现；至少 5 种「害」在叙事/选项上生效。 |
| M2 物证与线索    | 物证栏、线索栏；凭物感应、语义解锁在 1–2 个剧情点可玩。 |
| M3 多界与多结局  | 2 个以上界可完整走通；至少 3 个结局条件实现并可触发。 |
| M4 可发布候选    | 存读档、主界面与各框齐全、核心结局与败北结局覆盖、无明显逻辑穿帮。 |

---

## 5. 风险与应对

- **AI 不稳定**：用「骨架强约束 + 最小可接受描述」定义通过标准；关键节点可保留策划兜底文案。
- **内容膨胀**：先做「最小可玩界」和「最小结局集」，再按优先级加界/结局。
- **进度评估**：叙事与 AI 调优难以精确估时，建议用「范围弹性、时间大致固定」的迭代，每轮砍非必要需求保可玩。

---

## 6. 小结

采用**轻量 Agile**：短周期迭代、以可玩版本为验收、内容与配置全部进 Git、AI 指令与测试用例版本化。这样既能应对叙事与 AI 的不确定性，又便于小团队协作和后期维护。若后续团队扩大，再考虑引入更正式的 sprint 节奏和角色分工即可。
