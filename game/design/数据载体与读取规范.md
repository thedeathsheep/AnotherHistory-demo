# 数据载体与读取规范

> 序章及后续章节的数据格式、文件组织、读取方式。  
> **AI 生成内容**的流水线与存储见《AI 流水线与存储规范》。

---

## 一、数据载体形式

### 1.1 文件组织

```
public/data/
├── manifest.json      # 章节顺序与入口配置（可选）
├── prologue.json      # 序章（独立文件）
├── skeleton.json      # 主内容（折戟原等，可保留或拆分为 zhejiyuan.json）
└── chapters/          # 或：按章节分目录（可选扩展）
    ├── prologue.json
    └── zhejiyuan.json

generated/             # AI 流水线输出（细纲、节点、文本），合并后复制到 public/data
└── chapters/{id}/
    ├── outline.json   # AI-1 输出
    ├── nodes.json     # AI-2 输出
    └── texts.json     # AI-3 输出
```

**推荐**：`prologue.json`、`skeleton.json` 由 AI 流水线合并生成后写入 `public/data`；loader 按 manifest 或默认顺序聚合。

### 1.2 单章节 JSON 结构（与 types 对齐）

```json
{
  "id": "Prologue",
  "name": "序章",
  "entry_node": "Prologue_01",
  "nodes": [
    {
      "node_id": "Prologue_01",
      "plot_guide": ["驿亭", "枯井", "书箱", "墨迹"],
      "taboo": [],
      "objective": "理解当前处境，决定是否靠近枯井",
      "description": "叙事文本...",
      "choices": [
        {
          "text": "选项文案",
          "next": "Prologue_02",
          "state": { "ming_zhu": -5 },
          "conclusion_label": null
        }
      ]
    }
  ]
}
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | string | ✓ | realm 唯一标识 |
| name | string | ✓ | 显示名称（如「序章」） |
| entry_node | string | ✓ | 入口节点 id |
| nodes | Node[] | ✓ | 节点列表 |
| **Node** | | | |
| node_id | string | ✓ | 节点唯一标识 |
| plot_guide | string[] | 可选 | 核心剧情导向（原 truth_anchors，策划自由加入） |
| taboo | string[] | 可选 | 禁忌 |
| objective | string | 可选 | 目标 |
| description | string | ✓ | 兜底/骨架描述 |
| choices | Choice[] | ✓ | 选项 |
| **Choice** | | | |
| text | string | ✓ | 选项文案 |
| next | string | ✓ | 后继节点 id，`__结案__` 表示结案 |
| state | StateDelta | 可选 | 三相增减 |
| conclusion_label | string | 可选 | 结案时异史结论标签 |

**兼容**：`plot_guide` 与 `truth_anchors` 二选一或并存；loader 读取时优先 `plot_guide`，无则用 `truth_anchors`。

## 二、读取形式

### 2.1 聚合模式（推荐）

1. **loadSkeleton()** 扩展为：
   - 读取 `manifest.json`（若有）得到章节列表 `["prologue", "skeleton"]`
   - 按顺序 fetch `prologue.json`、`skeleton.json`
   - 每个文件为单个 **Realm** 结构
   - 合并为 `Skeleton { realms: [prologueRealm, zhejiyuanRealm, ...] }`
   - 返回聚合后的 Skeleton

2. **无 manifest**：默认加载 `["prologue.json", "skeleton.json"]`，若某文件 404 则跳过。

### 2.2 单文件模式（兼容）

- 保留现有逻辑：仅加载 `skeleton.json`，其中 `realms` 已含序章与主内容。
- 序章可作为 `realms[0]` 写入 `skeleton.json`，无需多文件。

### 2.3 接口约定

```ts
// 单章节文件结构
export interface RealmData {
  id: string
  name: string
  entry_node: string
  nodes: NodeData[]
}

export interface NodeData {
  node_id: string
  plot_guide?: string[]   // 优先
  truth_anchors?: string[] // 兼容
  taboo?: string[]
  objective?: string
  description: string
  choices: Choice[]
}
```

## 三、序章与主线的衔接

- 序章 realm 的结案节点 `next: "__结案__"` 时，`currentNodeId` 置空，进入「结案/归档」状态。
- 游戏逻辑检测到结案后，可：
  - 展示卷轴、异史；
  - 提供「进入下一界」按钮，调用 `startRealm("ZheJiYuan")`；
  - 或返回主菜单选择下一章。
